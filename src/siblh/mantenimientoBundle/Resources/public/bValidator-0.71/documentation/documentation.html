What is bValidator?

bValidator is a jQuery plug-in for client side form validation. It supports fully customized error messages defined with template and CSS, many validation functions (actions) like date, email, URL, IP address, validation of dynamic forms, customized validation functions and more.

Home page: http://code.google.com/p/bvalidator/

Setup

For a basic setup you will need to include into your web page which contains form you want to validate jquery.bvalidator.js file or (recommended) compressed file with YUI compressor jquery.bvalidator-yc.js and, of course, jQuery:

include javascript:

<script type="text/javascript" src="jquery-1.7.2.min.js"></script>
<script type="text/javascript" src="jquery.bvalidator.js"></script>

or instead of jquery.bvalidator.js include jquery.bvalidator-yc.js which is the same file but compressed and much smaller in size:

<script type="text/javascript" src="jquery.bvalidator-yc.js"></script>

You will also need to include CSS file with styles for message tooltips:

include CSS:

<link href="bvalidator.css" rel="stylesheet" type="text/css" />

The web page should have elements you want to validate. This is usually a form, but validator can also work on input fields which are not inside the form (see example). Note the data-bvalidator input field attribute which tells bValidator which validation functions (actions) to invoke for the field. The name of that attribute can be changed and this can be for example class or rel, see options for details.

HTML:

<form id="myform">
	Email: <input type="text" data-bvalidator="required,email"/>
	Url: <input type="text" data-bvalidator="url"/>
</form>

And, at last, you need to initialize bValidator for the form:

javascript:

<script type="text/javascript">
	$('#myform').bValidator();
</script>

This code should be inserted after the form or you can use jQuery's $(document).ready to execute the script as soon as the webpage's Document Object Model (DOM) has been loaded. In that case this javascript code can be before or after the form:

<script type="text/javascript">
	$(document).ready(function () {
		$('#myform').bValidator();
	});
</script>

Configuration options

bValidator can be configured by passing object with options to the validator constructor. Passed options will override default options.

<script type="text/javascript">
	
	var options = {
		singleError: true,
		showCloseIcon: false
	};

	$('#myform').bValidator(options);

</script>

OR by defining a global object with options called bValidatorOptions. bValidatior will check if this object exists and if does, will load options from it and override default options.

<script type="text/javascript">
	
	var bValidatorOptions = {
		singleError: true,
		showCloseIcon: false
	};

	$('#myform').bValidator();

</script>

OR BOTH, you can define a global bValidatorOptions object and also pass options to the validator constructor. In that case options from the bValidatorOptions object override default options and options passed as argument to constructor will override options from the bValidatorOptions object.

This option tells which modifiers to automatically call for some validation action. By default trim modifier is called for stated actions. So you don't have to explicitly put modifier into data-bvalidator-modifier attribute. You can put more auto modifiers for one action and use parameters just like in the attribute. For example:
'myValidator': ['trim', 'myMod', 'myMod2[1:3]']
ajaxAnswerValid 'ok' Valid response for ajax validation with the ajax action. If value is valid ajax request should return this string.

By default, if there are more validation actions in data-bvalidator attribute they are all invoked when validation is triggered.
If you set this option to true actions will be invoked till value is invalid by one action and check by rest of the actions will be skipped. For example:
data-bvalidator="number,max[10]"
if check with number action fails max action will not be called.

Callbacks

Callbacks are user defined functions that are called by validator in a certain moment. Callbacks are defined the same way as configuration options. Here is an example of defining callback:

<script type="text/javascript">
	
	var options = {
		onBeforeValidate: function(element, action){
			// do something here
		},
	};

	$('#myform').bValidator(options);

</script>
Callback Arguments Description
onBeforeElementValidation
  1. jQuery object containing input field
 Called before the validation routine starts for each element (input field) that is validated. Returning false will cause the validation to be skipped for that element.
onAfterElementValidation
  1. jQuery object containing input field
  2. array containing all error messages for the element
 Called after the validation routine ends for each element (input field) that is validated.
If false is returned error messages (if any) won't be displayed, and binding events (validateOn, errorValidateOn) to input fields will be skipped.
Returning 0 (zero) will only disable bValidator error messages. See example 10.
onBeforeAllValidations
  1. jQuery object containing all inputs that will be validated
 Called once before the validation is started. Returning false will skip the validation. For example, if you validate a form with more input elements this callback will be called once before all validations and if false is returned the form will not be validated at all.
onAfterAllValidations
  1. jQuery object containing all inputs that are validated
  2. validation result (true or false)
 Called once after all validations are done. For example, if you validate a form with more input elements this callback will be called once after all validations are done and error messages (if any) are displayed. See example 10.
onBeforeValidate
  1. jQuery object containing input field
  2. string validator action (example: min[5])

Called before the validation routine starts for each validation action. Returning false will cause the validation to be skipped for that validator action.
This callback function will be called for each validation action so if you put more actions on one form element for example: max[10],email,required this callback will be called two times (for max and for email).
onAfterValidate
  1. jQuery object containing input field
  2. string validator action (example: max[10])
  3. validation result (true or false)
 Called after each validation routine ends. Returning false will stop executing that validator action, displaying error messages and binding events (validateOn, errorValidateOn) to input fields will be skipped.
Will be called for each validation action (see onBeforeValidate description).
onValidateFail
  1. jQuery object containing input field
  2. string validator action (example: max[10])
  3. array containing all error messages for the element
 Called after validation fails.
Will be called for each validation action that fails (see onBeforeValidate description).
onValidateSuccess
  1. jQuery object containing input field
  2. string validator action (example: max[10])
 Called after validation is sucessfull.
Will be called for each validation action that is successful (see onBeforeValidate description).

Error messages and localization

bValidator can retrieve error message from:

  • data-bvalidator-msg attribute
  • configuration options object passed as an argument to constructor
  • global bValidatorOptions object
  • data-bvalidator-msg-actionName attribute. Replace actionName with the name of validation action (see example).

Most of the validator actions has error message named same as the action in configuration object. If you use some custom validation function (example) you can also add error message named same as the validation function into the configuration object.
You can use {0}, {1}, {2}, {3},... tags in error messages and they will be replaced with validator action parameters in [].

Error messages can be localized (translated). To translate messages you need to pass translated messages as options to validator constructor:

<script type="text/javascript">
	
	var options = {
		lang: 'hr',
		errorMessages: {
			hr: {
				'default':    'Ispravite ovu vrijednost.',
				'equalto':    'Unesite ponovno istu vrijednost.',
				'differs':    'Unesite različitu vrijednost.',
				....
			}
		}
	};

	$('#myform').bValidator(options);

</script>

OR define global bValidatorOptions object with translated error messages that will be used in all validator instances:

<script type="text/javascript">
	
	var bValidatorOptions = {
		lang: 'hr',
		errorMessages: {
			hr: {
				'default':    'Ispravite ovu vrijednost.',
				'equalto':    'Unesite ponovno istu vrijednost.',
				'differs':    'Unesite različitu vrijednost.',
				....
			}
		}
	};

	$('#myform').bValidator();

</script>
You can see all default English error messages in jquery.bvalidator.js file (search for first occurrence of 'errorMessages' inside the file). If you made translation please post it to bValidator discussion group. I will also put it on download page.

Validator actions

There are two types of validator actions: validation actions and modifying actions (from version 0.7).

Validation actions (validators) are functions that checks if a value is valid.

Modifying actions (modifiers) are functions that change the value of the field before validation. This is useful if you want to clean up user input for example to trim whitespaces before validation.

Validations which validator will perform on a single input field are stated by default in data-bvalidator attribute and modifiers are stated in data-bvalidator-modifier attribute.
You can use different attributes like class or rel if you need that your HTML is valid. Just set the name of the attribute in validateActionsAttr or modifyActionsAttr option.

Some actions can have parameters and those are stated in square brackets after the action name and delimited with ':'. For example: [param1:param2:param3]. You can have unlimited number of parameters per action.

For example:

<input type="text" data-bvalidator="maxlength[10]" data-bvalidator-modifier="trim" />
<input type="text" data-bvalidator="max[10],email,required" />
<input type="text" data-bvalidator="differs[someElementID]" />

 

Validation actions

Here are listed built-in validator actions but it is very simple to add custom validation functions (see example and getActions API function).

Validator Parameters Description
alpha   Checks if all characters are alphabetic. Empty space (' '), underscore(_), minus(-) and period (.) are also valid.
alphanum   Checks if all characters are alphanumeric. Empty space (' '), underscore(_), minus(-) and period (.) are also valid.
digit   Checks if all characters are digits.
number   Checks if value is a number.
email   Checks if value is a valid E-mail.
image   Checks if value ends with image file extension jpg, jpeg, png, gif or bmp. Can be useful for file fields.
url   Checks if value is a valid URL.
regex
  1. pattern
  2. modifier (optional)

Checks if value matches pattern given by parameter.
If you'll need to use characters like [],: in pattern which are also used by validator to parse action arguments you'll have to write custom validator function like in this example.
As second parameter regular expression modifier can be passed.
date
  1. date format
 Checks if value is a valid date in format specified by action parameter. Format can be any combination of mm, dd, yyyy with one character separator between. Example: 'mm.dd.yyyy' or 'yyyy-mm-dd'.
required  

Is a value required.

If you put required in front of other actions like data-bvalidator="required,max[10],email" then error message that field is required will be shown in error messages tooltip with other error messages.

If you put required after other actions like data-bvalidator="max[10],email,required" then error message that field is required will not be shown in error messages tooltip if there are some more error messages.

NOTE that if a field is not marked as required and it is empty it will not be validated at all. For example, if you have field with data-bvalidator="email" and you leave it empty, email action will not be called. If you want to call validator action even if value is empty and not required use valempty.
between
  1. minimum value
  2. maximum value

Checks is value between values in action parameters.

If the input is a checkbox and it is in a group of checkboxes (all with the same name for example name="chk[]") or select then between means "number of selected checkboxes (options) must be between values set by action parameters". See example.

If you use between with checkbox or select you should use custom error messages. See errorMessageAttr option.
max
  1. maximum value

Checks is value less or equel than action parameter value.

If the input is a checkbox and it is in a group of checkboxes (all with the same name for example name="chk[]") then max means "select no more then max checkboxes". See example.

If the input is a select then max means "select no more then max options".

If you use max with checkbox or select you should use custom error messages. See errorMessageAttr option.
min
  1. minimum value

Checks is value greater or equel than action parameter value.

If the input is a checkbox and it is in a group of checkboxes (all with the same name for example name="chk[]") then min means "select at least min checkboxes". See example.

If the input is a select then min means "select at least min options".

If you use min with checkbox or select you should use custom error messages. See errorMessageAttr option.
rangelength
  1. minimum length
  2. maximum length
 Checks if length of the value is between action parameters values.
maxlength
  1. maximum length
 Checks if length of the value is smaller than action parameter value.
minlength
  1. minimum length
 Checks if length of the value is greater than action parameter value.
differs
  1. element id
 The value has to be diferent from the value of field which ID is passed as parameter.
equalto
  1. element id
 The value has to be equel to the value of field which ID is passed as parameter.
extension

file extensions

 Checks if value ends with '.' followed by any of passed parameters.
You can put as many file extensions you need as parameters to this action. For example: extension[jpg:png:txt]
ip4   Checks if the value is a valid IPv4 address.
ip6   Checks if the value is a valid IPv6 address.
NOTE that this action uses complex regular expression which can be pretty slow and doesn't work in Chrome which shows "Aw, Snap!" page! So be careful where you use this action or make your own custom action for IPv6 address validation.
valempty   Validate field even if it's not marked as required and is empty. This action makes sense only if some other custom validator action is also used on that field and the field is not required. For example, you can have an input box that can be empty in some cases that depend on some other field. In that case you cannot put required action on it because it can be empty, but you need to validate it even if its empty. See example.
ajax
  1. URL for the ajax request

Action for server side validation with ajax. As the action argument put URL for the ajax request, but keep in mind that URL should not contain characters which bValidator uses to parse actions and parameters. Those are by default semicolon and comma (see paramsDelimiter and actionsDelimiter options). So use relative URLs or absolute from server root. For example:
data-bvalidator="ajax[/validate/checkusername],required".

bValidator tries to be smart and to optimize requests to the server:

  • It will use asynchronous and synchronous ajax calls depending on the situation. Sometimes it must use synchronous call for example before form submit.
  • bValidator will not make ajax request if a value is invalid by some other validator. But this depends on the actions order in data-bvalidator attribute. If you don't want to make ajax request if value is already invalid put the ajax action in the last place. For example:
    data-bvalidator="email,ajax[/url]"
  • It will not make ajax request if a value of the input is not changed from the last check with ajax. It will use cached result.

See also options regarding ajax and example.

API functions

When you make an instance of validator you can call some functions on that instance. You can access validator instance through jQuery data function:

<script type="text/javascript">
	
	$('#myform').bValidator();
	
	// check if form is valid
	if($('#myform').data('bValidator').isValid()){
		// do somenthing
	}

</script>

.data('bValidator') will return reference to the first instance of bValidator.
You can have more instances of bValidator on the same form (see example 12) and all references are returned by .data('bValidators'). Note the 's' here.

<script type="text/javascript">
	
	$('#myform').bValidator();
	$('#myform').bValidator(null, 'secondInstanceName');
	
	// validate form with the first instance
	$('#myform').data("bValidators").first.validate();

	// validate form with the second instance
	$('#myform').data("bValidators").secondInstanceName.validate();

</script>

NOTE: prior to version 0.6 you could get validator instance like this: 

<script type="text/javascript">

	var instance = $('#myform').bValidator();

</script>

From version 0.6 plugin function doesn't return validator instance to maintain chainability. Now you can do for example:

<script type="text/javascript">
	
	// bind validator on all forms on page
	$('form').bValidator();

	// or bind validator on all forms on page and set some css for the forms
	$('form').bValidator().css('width', '100%');

</script>

 

Function Arguments Description
getOptions   Returns the options object of the current validator instance.
isValid
  1. jQuery object containing inputs to check only
    (optional)
 Just checks if inputs in current validator instance are valid. Doesn't show error messages.
removeMsg
  1. jQuery object containing input
 Removes (hides) the message
showMsg
  1. jQuery object containing input
  2. string message
 Displays message. This function doesn't have anything with validation, it just displays tooltip message on the element specified with the first argument. Message can be displayed for any element on a page. Element doesn't have to be part of the form (or container element) for which bValidator is instanced.
getInputs   Returns all input fields associated with the validator.
bindValidateOn
  1. jQuery object containing inputs
This function is helpful when using validateOn option and dynamic forms. If you add some new input elements to the form after validator is already initialized and you want to validate them on event specified with validateOn option you need to call this method.
If you are not using validateOn option with dinamic forms you can forget about this function.
reset   Resets the validator. Hides all error messages like the validator is only initialized. If you have a reset button on your form this function is automatically bind to it.
destroy   Removes existing bValidator instance.
validate
  1. if true error messages are not displayed
    (optional)
  2. jQuery object containing inputs to validate only (optional)
 Does validation.
getValidators  

Returns the object with all built-in validator action functions of the current validator instance. This is useful for adding custom validator actions which are not accessible from global scope or overriding existing built-in actions. For example, you can add your validation function like this:

<script type="text/javascript">
	$('#myform').bValidator();
					
	$('#myform').data('bValidator').getValidators().myAction = 
		function(valueObject){
			// do something here and return true or false
			if(valueObject.value == 'valid')
				return true
			return false
		}
</script>
Note that parameter passed to validator functions is object with 'value' property which holds the input value. This is not the case with custom validator functions in global scope where the first parameter is input value.
getModifiers  

Returns the object with all built-in modifiers of the current validator instance. This is useful for adding custom modifier actions which are not accessible from global scope or overriding existing built-in actions. For example, you can add your modifier function like this:

<script type="text/javascript">
	$('#myform').bValidator();
					
	$('#myform').data('bValidator').getModifiers().myMod = 
		function(oldValue){
			// ... change oldValue
			return 'modifiedValue';
		}
</script>

See here how to access validator instances if you need to call some functions on them and also see themes section below for changing messages style. See also example 13.

13 - Just displaying messages

<span> element

In this example no validation is done. Just tooltip messages are displayed. This can be useful for example if you validate all your data on the server and just want to show messages to a user or want to show some other information in tooltip messages beside validation messages.
Note that messages can be displayed on any element, it doesn't have to be input and it doesn't have to be inside container element that bValidator is bind on. In this example more instances of bValidator are used for various styles of error messages (see also example 12).

14 - Using modifiers

Enter E-mail: (put some whitespaces in the beginning or the end)

Enter only digits: (try to enter some letters)

Enter URL: (try to enter without http://)

Enter some word: (put some whitespaces in the beginning or the end)

Modifiers are functions which purpose is to change the input value before validation. This can be useful to clean up data user has entered for example to strip whitespaces or similar. Modifiers are called before all validation actions. By default modifiers should be listed in data-bvalidator-modifier attribute just like validators in data-bvalidator attribute (see modifyActionsAttr option). For some validation actions trim modifier is automatically called (see autoModifiers option).

In this example on the first field trim modifier is automatically called because email action is mentioned by default in autoModifiers option.
On the second field custom modifier is called which will delete all but digits. Generally it is not good idea to use modifiers that change user input in that way, but it is here for the example.
The third field will add 'http://' if you enter URL without it and also automatically trim spaces.
The fourth field has only modifier action, no validation at all. You can use only modifier actions on the field without data-bvalidator attribute and validation.

Note that first argument of a custom modifier function will be a value of the input and other arguments (if any) are parameters for that action stated in square brackets after the action name in the data-bvalidator-modifier attribute. Custom modifier function should return a new value for the input field. If nothing (undefined) is returned the input value wont't be changed.
